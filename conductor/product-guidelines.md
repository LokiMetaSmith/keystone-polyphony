# Product Guidelines\n\n## Prose and Communication Style\n- **Tone:** Professional, direct, and collaborative. Avoid fluff and corporate jargon.\n- **Audience:** Senior software engineers, system architects, and autonomous AI agents.\n- **Clarity:** Prioritize technical rationale and intent over mechanical narration.\n\n## User Experience (UX) Principles\n- **Visual Feedback:** Distributed state changes (baton acquisition, network health) must be represented visually and in real-time.\n- **Minimalism:** Keep the dashboard focused on the "Pulse" of the network. Hide complexity behind drill-down views.\n- **Platform Native:** Use platform-native primitives (CSS Gradients, SVG, standard UI components) to ensure a modern, lightweight feel.\n- **Zero-Touch:** The bootstrapping process (`install.sh`, `./keystone-polyphony.sh`) must be seamless and idempotent.\n\n## Technical Standards\n- **Validation:** Use `Pydantic` for all data modeling and validation in Python.\n- **Security:**\n  - Never log, print, or commit sensitive credentials.\n  - Rely on cryptographic Ed25519 identity pairs for authentication.\n  - All mesh communication must be encrypted.\n- **Concurrency:** Prefer asynchronous programming (`asyncio`, `aiohttp`) for high-performance I/O operations.\n- **Testing:** Aim for exhaustive verification of distributed state logic (CRDTs, Vector Clocks) and network resilience.\n\n## Collaboration Norms\n- **The Baton:** Contributors must respect the explicit ownership of resources defined by the baton system.\n- **Stigmergy:** Leave digital markers (logs, thoughts, state updates) to help others (human or agent) understand the project progression.
